# TIP-008: Event-Driven Integration — Reducing Cross-Boundary Reasoning

**Status:** Recommended for AI-maintained systems with complex service interactions.

## Context

AI agents work best when they can reason locally.

The moment a business workflow depends on synchronous execution across multiple components, the reasoning scope expands.

To safely modify a feature, the agent must understand:

* the caller;
* the receiver;
* shared contracts;
* failure behavior;
* retry logic;
* execution ordering;
* side effects.

The business change is no longer local.

The agent must reason across boundaries.

As systems grow, this becomes one of the largest sources of architectural complexity.

We need Event-Driven Integration.

---

## The Problem

Traditional request-response communication creates hidden reasoning dependencies.

Consider a typical workflow:

```pseudo
Checkout
    → Payment Service
        → Fraud Service
            → Inventory Service
                → Notification Service
```

A seemingly simple business change can require understanding multiple execution paths.

The AI agent must answer:

* What happens if the payment fails?
* What happens if inventory is unavailable?
* Which service owns the business rule?
* Where is the final state transition performed?
* Which retries are safe?

The business workflow becomes distributed across multiple execution contexts.

---

## Event-Driven Integration

Event-Driven Integration separates execution into independent business reactions.

Instead of:

```pseudo
Checkout
    → Request Payment
    → Wait
    → Process Response
    → Continue
```

the workflow becomes:

```pseudo
Checkout
    → Publish PaymentRequested
```

and later:

```pseudo
PaymentProcessor
    → Publish PaymentCompleted
```

and later:

```pseudo
OrderHandler
    → Handle PaymentCompleted
```

Each actor owns its own responsibility.

Each actor can be understood independently.

---

## The Locality Principle

A component should not need to understand how another component performs its work.

It should only understand:

* the event it emits;
* the event it consumes;
* its own local business rules.

This dramatically reduces the amount of context required to modify behavior safely.

---

## Event Contracts

Events are contracts.

They should be:

* immutable;
* versioned;
* explicit;
* business-oriented.

Examples:

```pseudo
PaymentRequested
```

```pseudo
PaymentCompleted
```

```pseudo
InventoryReserved
```

```pseudo
OrderCancelled
```

Events should describe facts and intentions.

They should not expose implementation details.

---

## Event Choreography

In AI Oriented Architecture, event choreography is generally preferred over orchestration.

Orchestration creates central control points:

```pseudo
Coordinator
    → Service A
    → Service B
    → Service C
```

Choreography creates autonomous reactions:

```pseudo
Event A
    ↓
Handler A

Event B
    ↓
Handler B

Event C
    ↓
Handler C
```

The system becomes a collection of local decision makers rather than a centralized execution chain.

---

## Internal Events Matter Too

Event-Driven Integration is not limited to distributed systems.

Many benefits appear inside a single application.

For example:

```pseudo
CheckoutHandler
    → Publish OrderPlaced
```

```pseudo
InventoryHandler
    → Handle OrderPlaced
```

```pseudo
NotificationHandler
    → Handle OrderPlaced
```

This can be implemented using:

* InMemoryEventBus;
* local event dispatchers;
* actor frameworks;
* event streams;
* distributed event brokers.

The architectural principle remains the same.

The deployment model is secondary.

---

## Why It Matters

Event-Driven Integration improves:

* feature isolation;
* repository navigation;
* code graph locality;
* test isolation;
* ownership clarity;
* agent modification safety.

Most importantly:

it allows agents to work inside local business boundaries rather than reconstructing large execution chains.

The goal is not asynchronous execution.

The goal is local reasoning.

---

## Relationship to AI Oriented Architecture

AI Oriented Architecture favors boundaries that minimize the amount of context required for change.

Event-Driven Integration helps by reducing direct runtime coupling.

A handler should not need to understand the internal implementation of another handler.

It should only understand the event contract that connects them.

This keeps business logic local and boundaries explicit.

---

## What This Is Not

Event-Driven Integration does not mean:

* every system must become distributed;
* every interaction must be asynchronous;
* RPC should never exist;
* orchestration is always wrong.

Some workflows require:

* strong consistency;
* immediate responses;
* transactional guarantees.

In those cases, synchronous communication remains appropriate.

The principle is simple:

Prefer event boundaries when they reduce reasoning complexity without violating business requirements.

---

## Recommendations

For AI engineering teams:

* Prefer business events over direct service-to-service dependencies.
* Use immutable event contracts.
* Keep handlers focused on local responsibilities.
* Avoid centralized orchestration when choreography is sufficient.
* Separate event producers from event consumers.
* Favor explicit event names that reflect business meaning.
* Use internal event buses as aggressively as external ones.
* Evaluate integration patterns based on reasoning locality, not deployment style.

Stop making components understand each other's implementation details.

Let them communicate through facts.

Let them react through local logic.
