# TIP-007: Strict JSON Gateways — Stop Defensive Programming in Business Logic

**Status:** Recommended for cleaner business logic and safer AI-generated code.

## Context

AI coding agents become overly defensive when input contracts are unclear.

If a method receives loose JSON, partially validated objects, nullable fields, or ambiguous runtime structures, the agent starts generating protection code everywhere:

* null checks;
* fallback values;
* type checks;
* defensive parsing;
* deep optional-field handling;
* speculative error branches.

This creates a dangerous pattern.

Business logic becomes mixed with input sanitation.

The agent spends tokens protecting the code from invalid states instead of implementing business behavior.

We need Strict JSON Gateways.

---

## The Problem

Many systems allow dynamic input uncertainty to leak into core handlers.

A business method receives an object and still has to ask:

* does this object exist?
* are required fields present?
* are field types valid?
* are enum values known?
* are nested objects populated?
* do defaults need to be reconstructed?

When those questions remain open, the AI agent fills the gap with defensive boilerplate.

The result is code that is longer, noisier, harder to review, and easier to corrupt during generation.

---

## Strict JSON Gateways

A Strict JSON Gateway is a hard validation boundary at the edge of a feature, service, or actor.

Its job is simple:

1. Accept loose external input.
2. Validate it against a strict schema.
3. Reject malformed payloads immediately.
4. Convert valid input into typed DTO contracts.
5. Pass only valid DTOs into business logic.

The core rule:

> Loose JSON must die at the boundary.

Once the request crosses the gateway, business code should operate on valid, typed, non-ambiguous contracts.

---

## DTO = Contract

In AI Oriented Architecture, a DTO is not just a data bag.

A DTO is a contract.

It defines:

* required fields;
* allowed types;
* valid enum values;
* nullability rules;
* structural expectations;
* boundary guarantees.

Examples:

```pseudo
contract PromoRequest {
    required code: NonEmptyString
    required customer_id: CustomerId
}
```

Business code should not repeatedly revalidate what the gateway has already guaranteed.

---

## Defensive Bloat

Without a strict gateway, business logic often looks like this:

```pseudo
function apply_promo(request) {
    if request is null:
        return

    if request.code is missing:
        throw InvalidPromoCode

    code = trim(request.code)

    if code is empty:
        throw InvalidPromoCode

    if request.customer_id is missing:
        throw InvalidCustomer

    promo = database.find_promo(code)

    if promo is null:
        throw PromoNotFound

    apply(promo, request.customer_id)
}
```

Some of this is business logic.

Much of it is contract uncertainty.

The agent has to modify code inside a noisy execution path.

---

## Strict Contract Flow

With a strict gateway, malformed input is rejected before business execution begins:

```pseudo
gateway PromoRequestGateway {
    accepts json
    validates PromoRequestSchema
    returns PromoRequest
}
```

The business method receives a trusted contract:

```pseudo
function apply_promo(request: PromoRequest) {
    promo = database.find_promo(request.code)

    if promo not found:
        throw PromoNotFound

    apply(promo, request.customer_id)
}
```

The remaining validation is business validation, not technical sanitation.

That distinction matters.

---

## Perimeter Validation

Strict JSON Gateways push technical uncertainty to the perimeter.

Good candidates for perimeter validation:

* required fields;
* JSON shape;
* primitive types;
* enum membership;
* ID format;
* string normalization;
* basic range checks;
* schema version compatibility.

Good candidates for business logic:

* whether a promo exists;
* whether a customer is eligible;
* whether a discount can be combined;
* whether an account is allowed to perform an action;
* whether a workflow state transition is valid.

The gateway protects structure.

The business layer protects meaning.

---

## Implementation Examples

Strict gateways can be implemented with:

* Zod or Valibot in TypeScript;
* Pydantic in Python;
* Records and validation attributes in C#;
* Java Records and Bean Validation;
* Go structs with strict decoding and validation tags;
* JSON Schema with generated typed contracts.

The tool does not matter.

The boundary does.

---

## Why It Matters

Strict JSON Gateways improve AI-generated code because they reduce ambiguity.

The agent no longer needs to guess whether every field may be missing, malformed, or null.

It can work against a stable contract.

This improves:

* business-logic density;
* patch predictability;
* test clarity;
* DTO consistency;
* agent reasoning;
* code review quality.

Most importantly:

the agent writes less defensive infrastructure and more domain behavior.

---

## Relationship to AI Oriented Architecture

AI Oriented Architecture depends on local reasoning.

Strict JSON Gateways help by separating two concerns:

* input validity;
* business correctness.

When these concerns are mixed, every business method becomes harder for an agent to understand and modify.

When they are separated, the business path becomes smaller, cleaner, and safer to change.

---

## What This Is Not

Strict JSON Gateways do not mean ignoring production reality.

Systems still need:

* observability;
* error reporting;
* schema evolution;
* backward compatibility;
* safe handling of untrusted clients.

The point is not to pretend invalid input cannot exist.

The point is to keep invalid input from leaking into core business execution.

---

## Recommendations

For AI engineering teams:

* Treat DTOs as boundary contracts, not loose data bags.
* Reject malformed JSON at the gateway.
* Use non-nullable required fields for mandatory parameters.
* Keep technical validation outside business handlers.
* Keep business validation inside business logic.
* Avoid speculative fallback chains in core methods.
* Generate typed contracts from schemas where possible.
* Make input guarantees explicit before asking agents to modify business flows.

Stop forcing AI agents to write runtime shields inside your core code.

Lock down the entry contract at the boundary.

Let the agent work with valid business facts.
